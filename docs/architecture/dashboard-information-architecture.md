# Redou 控制台首页信息架构

Redou 控制台首页定位为任务运行控制台。它负责展示任务状态、任务选择、产物、变更摘要和诊断线索；prompt 执行、工具调用、LLM 调用、skills 和任务 skill 打包继续由 `vendor/hermes` 承担。

## 首页板块

首页固定围绕七个板块组织：

- 运行概览：展示活跃、排队、需处理、失败、完成、今日 tokens 和今日工具调用次数，并展示当前选中任务的“进入任务”入口。
- 任务总览：多任务列表和轻量运行指标，支持搜索，并按需处理、运行中、排队中、失败、最近完成、未开始、更早历史、已取消排序。
- 任务详情：由任务总览中选中的任务驱动，按状态展示运行阶段、工具、模型、耗时、错误、测试摘要和阶段时间线。
- 需要处理：只展示等待用户介入的事项，例如确认、权限、覆盖、失败待重试、路径异常、skill 冲突、测试失败和 provider/model 配置错误。
- 交付物：只展示真实产物，例如修改文件、生成文档、生成 skill、测试报告、导出包、运行报告、路径契约报告和冒烟测试报告。
- 变更预览：展示最近文件变更数量和文件列表，不在首页展开完整 diff。
- 诊断日志：弱化展示 Hermes 原始事件、工具日志和终端输出摘要，原始日志默认折叠。

## 任务状态定义

首页统一使用这些任务状态：

- 未开始
- 排队中
- 运行中
- 需处理
- 失败
- 已完成
- 已取消
- 暂停

前端通过 `normalizeTaskStatus(rawStatus)` 将旧状态归一化。`done`、`completed` 和 `success` 映射为已完成；`running` 和 `active` 映射为运行中；`queued` 和 `pending` 映射为排队中；`blocked`、`waiting`、`needs_attention` 和 `confirmation_required` 映射为需处理；`failed` 和 `error` 映射为失败；`cancelled`、`canceled` 和 `interrupted` 映射为已取消；`paused` 映射为暂停。

不再直接显示“空闲”。旧的 `idle` 只是后端运行快照的缺省值，无法表达用户真正关心的任务状态；首页会结合任务是否开始、是否完成、是否等待用户来映射为未开始、已完成或需处理。

## 总览与详情边界

任务总览只回答“有哪些任务、哪个最需要看、现在大概在做什么”。它使用紧凑任务卡，显示任务名、状态、当前或最后动作、耗时/时间、provider/model 简写、LLM 调用、工具调用、tokens 和产物数量。

任务详情回答“当前选中任务内部发生了什么”。它显示模型、工具、命令、耗时、最近事件、阶段时间线，并按运行中、已完成、需处理、失败、未开始/排队中提供不同字段。

## 阶段事件

如果任务事件中存在 `run_stage`，首页优先使用显式阶段事件展示阶段时间线。

如果没有显式阶段事件，首页只做轻量推断：读取/搜索类事件归入分析项目，写入/补丁/文件变更归入修改代码，测试命令归入测试验证，打包/导出命令归入打包输出，最终回复或完成事件归入整理结果。首页不显示精确百分比，只显示当前阶段和阶段时间线，并标注推断来源。

任务详情优先使用 Hermes `run_stage` 事件，缺失时使用 Redou 前端 fallback 推断。

## “进入任务”语义

“进入任务”表示把当前选中任务设为 Redou 当前任务，并进入任务工作区处理。当前桌面端没有独立任务详情路由时，该按钮复用已有 `setActiveChatTask` 和 `/chat` 工作区逻辑；如果跳转失败，首页给出轻量提示而不是崩溃。
